# 필드 엔지니어 역량·지식 비전 — "벤더 대표로서 엔지니어는 무엇을 할 수 있고 어디까지 알아야 하는가"

> 목적: "필드 엔지니어 1인 대체"의 실체를 **측정 가능한 역량(DO) × 지식 깊이(KNOW)** 로 정의한다.
> 이 문서는 AI가 채워가야 할 **역량 지도(WorkAtom taxonomy의 원천)** 이자, 어디까지가 사람의 몫인지의 경계선이다.

---

## 0. 핵심 프레임: 엔지니어는 사실 4개 역할의 융합

벤더는 "박스"를 파는 게 아니라 **한 명의 신뢰받는 사람이 전달하는 결과(outcome)** 를 판다. 그 사람이 곧 제품 경험이다.

| 역할 | 한 문장 | 무게중심 |
|------|---------|---------|
| **Presales SE** | 요구를 솔루션으로 설계하고 PoC로 증명해 수주 | 설계·판단·경쟁 |
| **Delivery Engineer** | 설치·구성·마이그레이션·인수 | 실행·정확성·안전 |
| **Ops/TAC Engineer** | 모니터링·장애대응·업그레이드·보안운영 | 진단·근본원인·복구 |
| **Embedded mini-PM** | 범위·일정·이해관계자·리스크 | 조율·커뮤니케이션 |

→ AI가 "필드 엔지니어를 대체"한다는 건 이 4개를 얼마나, 어느 깊이로 대신하느냐의 문제다.

## 1. DO 축 — 라이프사이클별 "엔지니어는 ~할 수 있어야 한다"

각 항목에 **[자동가능 / 하이브리드 / 사람전용]** 과 "AI가 이걸 하려면 무엇을 알아야 하는가"를 붙인다.

### Phase 0 — 발굴 & 설계 (Presales)
- 반쯤 형성된 고객 요구를 끌어낸다 ("제로트러스트 하고 싶다", "VMware 갱신이 비싸다") — *[하이브리드]* 대화·맥락 추론
- 비즈니스 동인 → 기술 요구 → 제품 역량으로 매핑 — *[자동가능]* 요구공학 + 제품지식
- **사이징**: 노드 수·처리량·스토리지 IOPS·동시 사용자·로그 보존 산정 — *[자동가능]* 제품별 Sizing Guide 공식(우리가 수집한 IAG/CC/HCI Sizing xlsx)
- 토폴로지 설계: 망분리·HA·DR·연동점 — *[하이브리드]* 인프라 아키텍처
- BOM + 라이선스 + 호환성 매트릭스 — *[자동가능]*
- **경쟁 분석**: HCI vs VMware/Nutanix, CC vs Darktrace/Vectra, EPP vs CrowdStrike, IAG vs Fortinet/Zscaler — *[자동가능]* 우리가 수집한 Battle Card·경쟁분석 PPT
- 설계 리스크 평가: SPOF·마이그레이션 위험·용량 헤드룸 — *[하이브리드]*

### Phase 1 — PoC / 검증
- 랩/PoC를 빠르게 세운다 — *[하이브리드]*
- **성공 기준을 고객과 합의** — *[사람전용]* (계약적 판단)
- 테스트 케이스 실행 → 증적 수집 → PoC 리포트 — *[자동가능]* (우리가 수집한 PoC test guide/checklist)
- 반론 대응·가치 시연 — *[사람전용]*

### Phase 2 — 딜리버리 / 구축
- **사이트 실사 & 준비도**: 전원·랙·냉각·네트워크 포트·IP 플랜·방화벽 룰·NTP/DNS·AD 도달성 — *[하이브리드]* 체크리스트 자동, 현장 확인은 사람
- **물리 설치**: 랙킹·케이블·전원 — *[사람전용, 영구]*
- **초기 구성**: 클러스터 init, 관리/스토리지/VM망, MTU/본딩, 스토리지풀, HA/DRS, 라이선스 — *[하이브리드→HCI는 API-first 자동화 목표]*
- **보안 하드닝 베이스라인**: 관리자 계정·MFA·인증서·감사로그·syslog — *[자동 검증/가이드, 적용은 사람]*
- **정책 구현** (제품별) — *[자문+사람적용]* (지금 우리 Service 1·2·3)
- **마이그레이션**: P2V/V2V·스토리지 이전·롤백 윈도우 있는 cutover — *[하이브리드, 고위험]*
- **연동**: AD/LDAP·RADIUS·SIEM/syslog·ITSM·기존 FW/proxy — *[자동가능, 레시피화]*
- SOW 대비 검증 & 인수 테스트 — *[자동 검증 + 사람 sign-off]*
- **as-built 문서 + 런북 + 인계 + 지식이전(KT)** — *[자동가능]* (우리 문서생성 강점)

### Phase 3 — 운영
- 일일 헬스(대시보드·알람·용량) — *[자동가능]* (지금 우리 진단)
- **업그레이드/패치**: 호환성 체크·단계 롤아웃·롤백 계획 — *[자문+사람실행]* (버전 델타 지식 핵심)
- 백업/DR 테스트 — *[하이브리드]*
- **장애대응 & RCA**: 로그·패킷·상관분석 — *[하이브리드]* (AI가 후보 제시, 사람 판단)
- 위협운영: 헌팅·알람 트리아지·SOAR — *[자문, critical 자동실행 금지]*
- 성능 튜닝 — *[하이브리드]*
- 주기 리뷰·최적화·갱신 — *[자동가능]*

## 2. KNOW 축 — 지식의 5개 층

| 층 | 무엇을 | "얼마나 깊이"의 기준 |
|----|--------|----------------------|
| **① 제품 마스터리** | 기능·설정·한계·버전·알려진 이슈 | "설정 위치"가 아니라 **왜·무엇에 영향·무엇이 깨지는지·기본값 vs 권장값 vs 하드닝값·설정 간 상호작용·버전 델타** |
| **② 인프라 펀더멘털** | 네트워크(L2/L3·VLAN·라우팅·NAT·VPN·MTU·LACP), 가상화/스토리지(HCI 내부·aSAN·HA/DRS·스냅샷), OS(Win/Linux) | 제품 아래의 원리 — 왜 MTU 불일치가 스토리지를 불안정하게 하는가 |
| **③ 보안 도메인** | 위협모델·MITRE ATT&CK·탐지로직·컴플라이언스(ISO27001/PCI/GDPR + **국내 K-ISMS-P·전자금융·개인정보보호법**) | 규제가 요구하는 로그 보존·감사·정책의 근거 |
| **④ 고객·비즈니스 맥락** | 산업·리스크 성향·규제 의무·기존 자산·조직 정치 | 같은 제품도 은행 vs 병원 vs 제조는 다르게 구성 |
| **⑤ 방법론·판단** | 트러블슈팅 규율·변경 안전·에스컬레이션 시점·불확실성 커뮤니케이션 | **자기 검증·blast radius 인식·자기 지식의 경계 인지** |

## 3. "어디까지 알아야 하나" — 깊이의 5개 임계선

엔지니어는 다음을 할 수 있을 만큼 알아야 한다:

1. **작업을 정확히 수행** (procedural)
2. **자기가 틀렸는지 안다 / 자기 작업을 검증** (self-check) ← 우리의 **read-back / false-pass 방지** 원칙
3. **인접한 것을 안 깨뜨린다** (blast radius 인식)
4. **고객 언어로 설명한다** (translation)
5. **자기 지식의 끝을 알고 에스컬레이션한다** (epistemic humility) ← 우리의 **INDETERMINATE** 원칙

> **4·5번이 "스크립트 실행기"와 "신뢰받는 엔지니어"를 가른다.**
> 자신만만하게 틀린 짓을 하는 AI는 쓸모없는 정도가 아니라 **위험**하다 — 그래서 우리가 INDETERMINATE는 절대 PASS로 안 세는 걸 코드로 박은 것이다. 이건 부수기능이 아니라 **"엔지니어급 신뢰"의 정의 그 자체**다.

## 5. 제품별 "반드시 알아야 하는 것" (구체)

- **HCI/SCP**: 클러스터 쿼럼, aSAN 리플리카/리빌드, MTU end-to-end, HA/DRS anti-affinity, 스냅샷 체인, VMware 마이그레이션 함정, **라이선스가 하드웨어 UUID에 바인딩**(노드 교체 시 재활성화)
- **IAG/SWG**: 인증 체인(AD/RADIUS/local), **SSL 디크립션 예외**(뱅킹/헬스 앱은 깨짐 → 분기별 리뷰), URL DB, DLP 파일타입, 802.1X, 컴플라이언스용 로그 보존
- **EPP**: 대규모 에이전트 배포, **monitor-then-enforce**로 EDR 베이스라인, 디바이스 컨트롤, hot-patch, **OS 손상 회피**, 예외 관리
- **CC/NDR/XDR**: 센서 배치, 이벤트소스 온보딩, **NTP(상관분석 품질의 근간)**, 알람 튜닝(노이즈 감소), **SOAR 플레이북 안전성**(격리/차단은 critical), 런북 링크
- **NGFW**: 정책 순서, NAT, VPN, 위협방지 프로파일

## 6. 엔지니어가 "대표"하는 것 (벤더 렌즈)

- **신뢰·브랜드**: 모든 행동이 Sangfor의 평판
- **책임(accountability)**: 인수 서명, 결과 소유, 법적 책임
- **관계**: 고객이 새벽 2시에 전화하는 그 사람

→ 그래서 AI는 **비가역·고객대면 행위에 대해 "자문 + 사람 sign-off"** 여야 하고, **감사추적·근거인용·면책**이 장식이 아니라 필수인 것. (지금 우리 리포트가 이걸 담고 있음)

## 8. 종합 아이디어 (한 문장)

> **"필드 엔지니어 대체" = KNOW(제품 전반 L2 폭 / 자문 L3 깊이를, 인용·버전관리·자기검증되는 지식으로) + DO(라이프사이클 전체를 read-only로 관통: 발굴→설계→검증→진단→문서)를 갖춘 AI를 만들되, 사람은 "비가역의 손"과 "책임의 서명"을 쥔다.**

**AI가 사람을 이기는 지점:** 버전 델타를 절대 안 잊고, 체크리스트 항목을 절대 안 빠뜨리고, 항상 근거를 인용하고, 모르는 걸 항상 표시한다.
**AI의 영구 한계:** 서버를 랙에 못 꽂고, 법적 책임에 서명 못 하고, 고객의 리스크 결정을 대신 못 한다.

이 두 문장이 우리 로드맵의 나침반이다 — **AI의 초능력을 극대화하고, 영구 한계는 사람에게 명확히 남긴다.**

## 9. 이 비전이 지시하는 다음 빌드 (우선순위)

1. **WorkAtom 카탈로그** — §1의 DO 항목을 제품×라이프사이클 매트릭스로 데이터화(자동/하이브리드/사람 라벨 + humanReason). 이게 "1인 대체율"의 정직한 분모.
2. **사이징 엔진** — 수집한 Sizing Guide(IAG/CC/HCI xlsx)를 규칙으로 → 요구 입력 → BOM/노드/라이선스 산출. (가장 빠른 Presales 자동화 승리)
3. **버전 델타/known-issues 지식** — RAG를 "이 버전에서 바뀐 것/알려진 이슈"로 구조화 → 업그레이드 자문의 근간.
4. **경쟁분석 봇** — 수집한 Battle Card로 "vs VMware/CrowdStrike/Darktrace" 자문.
5. **연동 레시피** — AD/LDAP/SIEM/syslog 표준 연동을 레시피화(Service 1 확장).
6. **RCA 후보 엔진** — 진단(Service 3)을 "증상→가능한 근본원인 후보 + 확인 단계"로 확장.
